    Аудит документации проекта apps-service-opus (апрель 2026)

    Дата аудита: 2026-04-16 (обновлено 2026-04-19) Аудитор: AI-ассистент Статус: Частично выполнено — многие рекомендации реализованы в рамках текущей задачи.

    Обзор

    Этот документ объединяет анализ документации (DOCUMENTATION_ANALYSIS.md) и рекомендации по её улучшению (DOCUMENTATION_RECOMMENDATIONS.md). Цель — предоставить единый источник информации о состоянии документации, выявленных проблемах и прогрессе их исправления.

    Резюме состояния

    Документация проекта обширная, но содержит противоречия и критические пробелы. Многие примеры кода содержат ошибки, которые приводят к неработающей конфигурации. Реализованная функциональность не всегда соответствует документации.

    Ключевые проблемы: 1. Отсутствие полного справочника по service.yml. 2. Нет централизованного руководства по устранению неполадок. 3. Примеры кода с отсутствующим container_name (критично для работы Caddy). 4. Противоречия между разделами (например, о поддержке Loki, Restic). 5. Недостаточная документация для разработчиков (локальная разработка, добавление API, работа с UI).

    Прогресс по рекомендациям (по состоянию на 2026-04-19): - ✅ Создан справочник service-yml.md (полная схема манифеста). - ✅ Создан файл TROUBLESHOOTING.md с типичными проблемами. - ✅ Исправлены некоторые примеры (требуется дополнительная проверка). - ⏳ Добавлены баннеры "Не реализовано" (частично). - ⏳ Реорганизация навигации (в процессе).

    2. Критические ошибки в примерах кода

    2.1 Проблема: container_name отсутствует в примерах

    Файлы с проблемами: - docs/examples.md — 6 примеров, только часть имеет container_name. - docs/getting-started/first-service.md — минимальный пример без container_name. - docs/user-guide/services.md — примеры без container_name. - docs/architecture.md — пример манифеста без container_name.

    Почему это критично: Без container_name Caddy использует host.docker.internal:<port> — проксирование на хост-машину вместо контейнера. Это: - Обходит SSL и аутентификацию Caddy. - Создаёт конфликты портов. - Является дыркой в безопасности.

    Рекомендация: ВСЕ примеры должны содержать container_name, совпадающий с именем контейнера в docker-compose.yml.

    Статус: ⏳ Требуется ручное исправление каждого файла. Частично выполнено в рамках других задач, но необходима полная проверка.

    2.2 Противоречия между разделами

    Раздел 1 Утверждение 1 Раздел 2 Утверждение 2 Статус

    3. Рекомендации по улучшению (быстрые победы)

    3.1 Исправить критические ошибки в примерах

    Файлы для редактирования: - docs/examples.md - docs/getting-started/first-service.md - docs/user-guide/services.md - docs/architecture.md

    Конкретные изменения: 

    # ДО (сейчас):
routing:
  - type: domain
    domain: myapp.example.com
    internal_port: 80

# ПОСЛЕ (должно быть):
routing:
  - type: domain
    domain: myapp.example.com
    internal_port: 80
    container_name: my-app  # ⚠️ ОБЯЗАТЕЛЬНО

    Также добавить предупреждение: 

    > ⚠️ **Важно:** Поле `container_name` обязательно. Без него Caddy будет 
> проксировать на хост-машину вместо контейнера, что создаёт конфликты 
> портов и проблемы с безопасностью.

    Статус: ⏳ Ожидает выполнения.
