# Ревью инвертированного diff (phase_3_failed.diff)

## Обзор изменений

Инвертированный diff показывает изменения, которые были **отменены** в текущем состоянии репозитория. Это означает, что данные изменения были удалены из кода.

## Основные изменения по файлам

### 1. Dockerfile (`_core/master/Dockerfile`)
**Удалено:**
- Пакет `rsync` из установки apt
- Установка Kopia CLI из GitHub releases
- Команда `kopia --version`

**Анализ:**
- Упрощение Docker образа за счет удаления Kopia
- Уменьшение размера образа и зависимостей
- Потеря функциональности Kopia для бэкапов

### 2. API endpoints бэкапов (`_core/master/app/api/routes/backups.py`)
**Крупные изменения:**
- Удалены сложные Pydantic модели (`BackupRequest`, `BackupSnapshotResponse`, `BackupOperationResponse`)
- Добавлены упрощенные модели (`BackupResponse`, `BackupCreateRequest`)
- Изменена структура endpoint'ов:
  - `/api/backups/{svc}/backup` → `/api/backups/service/{service_name}/backup`
  - `/api/backups/{svc}` → `/api/backups/service/{service_name}`
  - `/api/backups/{svc}/restore/{snapshot_id}` → `/api/backups/service/{service_name}/restore`
  - `/api/backups/snapshot/{snapshot_id}` → `/api/backups/{backup_name}`

**Функциональные изменения:**
- Удалена проверка `backup_config.enabled`
- Удален dry-run режим
- Упрощена логика восстановления (по ID вместо snapshot_id)
- Удалено управление retention днями
- Упрощена обработка ошибок

### 3. Основной файл приложения (`_core/master/app/main.py`)
**Изменения:**
- Добавлен `BackupManager` вместо `KopiaBackupManager`
- `app.state.backup` теперь ссылается на `BackupManager`
- `app.state.kopia_backup` становится основным Kopia менеджером
- Удалена проверка legacy таблиц бэкапов

### 4. KopiaBackupManager (`_core/master/app/services/kopia_backup_manager.py`)
**Удалено:**
- Метод `list_snapshots()` для работы с БД
- Метод `restore_snapshot()` с полной логикой восстановления
- Метод `delete_snapshot()` с удалением из Kopia и БД
- Параметр `subprocess_timeout`
- Вся логика работы с таблицей `BackupRecord`

**Осталось:**
- Базовая логика создания бэкапов через Kopia
- Проверки паролей и репозиториев
- Упрощенная логика работы с Kopia CLI

### 5. Тесты API endpoints (`_core/master/tests/unit/test_backup_endpoints.py`)
**Упрощение:**
- Удалены интеграционные тесты с моками Kopia
- Оставлены только тесты Pydantic моделей
- Удалена сложная логика тестирования endpoint'ов

### 6. API клиент платформы (`_core/platform-cli/apps_platform/api_client.py`)
**Удалено:**
- Методы для работы с Kopia API:
  - `create_backup()`
  - `list_backups()`
  - `restore_backup()`
  - `delete_backup()`
- Вся логика retry при сетевых ошибках
- Сложная обработка ошибок клиента

**Добавлено:**
- Упрощенный API для работы с новыми endpoint'ами

### 7. CLI команды (`_core/platform-cli/apps_platform/cli.py`)
**Удалено:**

  - `backup create`

**Добавлено:**
- Упрощенная команда `backup` с прямым вызовом API

### 8. Тесты API клиента (`_core/platform-cli/tests/test_api_client.py`)
**Полностью удален файл** - все тесты для старого API клиента

### 9. Тесты CLI (`_core/platform-cli/tests/test_cli.py`)
**Удалено:**
- Все тесты для команд бэкапов Kopia
- Моки для KopiaBackupManager
- Тесты проверки конфигурации сервисов

## Анализ изменений

### Положительные стороны:
1. **Упрощение архитектуры**: Удаление сложной логики Kopia делает систему проще
2. **Уменьшение зависимостей**: Удаление rsync и Kopia CLI уменьшает размер образа
3. **Более простой API**: Новые endpoint'ы проще для понимания и использования
4. **Упрощенная обработка ошибок**: Меньше сложных проверок и условий

### Отрицательные стороны:
1. **Потеря функциональности**: 
   - Нет dry-run режима
   - Удалено управление retention политиками
   - Нет проверки включения бэкапов в конфигурации сервисов
2. **Потеря интеграции с БД**: 
   - Удалена таблица `BackupRecord`
   - Нет истории бэкапов в базе данных
   - Нет статусов операций
3. **Упрощение CLI**: 
   - Меньше команд управления
   - Нет детального контроля над бэкапами
4. **Удаление тестов**: 
   - Потеря покрытия для сложной логики
   - Риск регрессий

### Риски:
1. **Без dry-run**: Пользователи не могут протестировать бэкап без реального выполнения
2. **Нет проверки конфигурации**: Сервисы могут запускать бэкапы даже если они отключены в конфигурации
3. **Упрощенное восстановление**: Восстановление по ID вместо snapshot_id менее надежно
4. **Потеря метаданных**: Нет информации о размере, статусе, retention днях

## Рекомендации

### Что можно вернуть:
1. **Dry-run режим**: Полезен для тестирования без реального создания бэкапов
2. **Проверка конфигурации**: Защита от случайных бэкапов отключенных сервисов
3. **Базовую историю операций**: Хотя бы в памяти приложения

### Что оставить упрощенным:
1. **API структуру**: Новые endpoint'ы более логичны
2. **Удаление Kopia CLI**: Если используется другая система бэкапов
3. **Упрощенные модели**: Если не нужна сложная метаинформация

### Тестирование:
1. **Добавить базовые тесты** для нового API
2. **Проверить интеграцию** с новой системой бэкапов
3. **Тестировать edge cases** восстановления и удаления

## Вывод

Изменения представляют собой **значительное упрощение** системы бэкапов с потерей части функциональности. Это может быть оправдано, если:
1. Kopia заменена другой системой бэкапов
2. Сложность старой системы была избыточной для требований
3. Нужна более простая и быстрая реализация

Однако стоит оценить, не потеряна ли критически важная функциональность для пользователей системы.
